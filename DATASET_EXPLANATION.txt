GreenChem Navigator - Dataset Explanation
=========================================

This file explains the machine learning dataset used in GreenChem Navigator.
It is written for a pitch/demo context, so it explains the dataset in simple
language without going too deep into code.


1. Dataset File
---------------

The dataset is stored in:

app/ml/training_data.csv

This dataset is used to train the prototype compatibility model. The goal of
the model is to predict whether a solvent can be considered compatible with a
given catalyst, catalyst role, and temperature condition.


2. Dataset Size
---------------

The dataset contains:

- 7,800 rows
- 28 columns in the CSV file

Important note:

The column "temperature_c" appears twice in the CSV. One copy is part of the
case description, and one copy is also used as a machine learning feature.
So technically the file has 28 columns, but 27 unique column names.


3. How The Rows Are Generated
-----------------------------

The dataset is generated from the local knowledge base:

app/knowledge_base.json

The knowledge base contains:

- 65 solvents
- 10 catalysts
- 20 catalyst roles in total
- 6 tested temperature values

The temperatures used are:

- 0C
- 25C
- 40C
- 60C
- 90C
- 120C

The number of rows is calculated like this:

65 solvents x 20 catalyst roles x 6 temperatures = 7,800 rows

Each row represents one possible chemical situation:

"This solvent + this catalyst + this catalyst role + this temperature"

Then the backend compatibility rules decide if that situation is compatible or
not compatible.


3.1 Source Of The Dataset
-------------------------

This dataset is not downloaded as a ready-made CSV from Kaggle or another
single public dataset.

It is generated by the project itself from the local knowledge base:

app/knowledge_base.json

The knowledge base contains curated information about:

- solvents
- catalysts
- catalyst roles
- GHS hazard codes
- solvent guide colors
- boiling points
- known incompatibilities
- reported replacement examples

The training CSV is generated by this script:

app/ml/dataset.py

The model training script is:

app/ml/train_model.py

The source of the labels is the internal compatibility logic:

app/compatibility.py

This means every row is created from:

solvent data + catalyst data + catalyst role + temperature + compatibility rules

The column "label_source" shows this:

curated_rule_bootstrap

This means the labels are generated from curated rules and the local knowledge
base. They are useful for a prototype and demo, but they are not experimental
laboratory labels.


3.2 External Data Sources Used In The Knowledge Base
---------------------------------------------------

Some solvent metadata can be expanded using public PubChem data through:

app/tools/import_pubchem_solvents.py

PubChem is used for public chemical metadata when available, such as:

- PubChem CID
- molecular formula
- molecular weight
- canonical SMILES
- hazard-code text when available

The project also uses curated solvent guide categories inspired by common green
chemistry solvent selection guides, such as CHEM21-style green/yellow/red
ratings.

In the knowledge base, the field "chem21_color" is used to represent the
solvent guide color:

- green = preferred or greener option
- yellow = usable but with concerns
- red = problematic or discouraged
- unknown = not enough local guide information

Boiling points, incompatibility notes, catalyst roles, and replacement examples
are curated inside the local JSON knowledge base for this prototype.

So the dataset source can be summarized as:

1. Local curated knowledge base
2. Internal compatibility rules
3. Optional PubChem public metadata
4. Green chemistry solvent guide categories

This design was chosen because our project needs an explainable prototype
dataset that is directly connected to the recommendation engine.


4. What The Model Predicts
--------------------------

The model predicts the column:

is_compatible

This is the target column.

Possible values:

- 1 = compatible
- 0 = not compatible

The model does not prove that a chemical reaction will work in a real lab.
It only predicts compatibility based on the local knowledge base and internal
chemical rules.

The final decision must always be made by a user or qualified chemical expert.
Wet lab validation is required.


5. Main Columns In The Dataset
------------------------------

The first columns describe the chemical case:

solvent
  The solvent being tested.
  Example: Acetone, Water, 2-MeTHF, Ethanol.

catalyst_formula
  The catalyst used in the process.
  Example: H2SO4, HCl, NaOH, Pd/C.

catalyst_role
  The role of the catalyst in the reaction.
  Example: acid_catalyst, base_catalyst, strong_acid.

temperature_c
  The reaction temperature in Celsius.
  Example: 25, 60, 90.


6. Target And Explanation Columns
---------------------------------

is_compatible
  This is the answer the model learns to predict.
  1 means compatible.
  0 means not compatible.

label_source
  Explains where the label came from.
  In this prototype, the labels come from curated rule-bootstrap logic.

label_reason
  Explains why the row was marked compatible or not compatible.
  This makes the dataset easier to understand and more transparent.


7. Machine Learning Feature Columns
-----------------------------------

These columns are used by the model to make predictions.

boiling_point_c
  The boiling point of the solvent.
  This matters because a solvent with a very low or very high boiling point can
  affect volatility, removal, heating, and process safety.

temperature_c
  The reaction temperature.
  Temperature matters because some incompatibilities only become important
  under heat.

chem21_rank
  A numeric version of the CHEM21 solvent guide color.

  green = 0
  yellow = 1
  red = 2
  unknown = 3

  This helps the model understand whether a solvent is preferred, usable, or
  problematic from a green chemistry perspective.

solvent_ghs_count
  The number of GHS hazard codes linked to the solvent.
  More hazard codes usually means more safety concerns.

solvent_incompatibility_count
  The number of known incompatibilities for the solvent.
  This helps the model detect risky solvent situations.

catalyst_incompatibility_count
  The number of known incompatibilities for the catalyst.
  Some catalysts are dangerous or reactive with certain solvent classes.

is_acid_role
  1 if the catalyst role is acidic.
  0 otherwise.

  This is useful because acid conditions can affect solvent stability.

is_base_role
  1 if the catalyst role is basic.
  0 otherwise.

  This is useful because base conditions can cause hydrolysis or other
  compatibility issues.

catalyst_supports_role
  1 if the catalyst actually supports the selected role.
  0 otherwise.

  Example:
  H2SO4 supports acid_catalyst, but it does not support base_catalyst.

is_heated
  1 if the temperature is 50C or above.
  0 if the temperature is below 50C.

  This helps the model separate room-temperature conditions from heated
  conditions.


8. GHS Hazard Code Columns
--------------------------

The dataset also contains binary columns for important GHS hazard codes.

Each column answers this question:

"Does this solvent have this hazard code?"

The value is:

- 1 = yes
- 0 = no

The columns are:

has_H225
has_H301
has_H302
has_H311
has_H312
has_H315
has_H319
has_H331
has_H332
has_H336
has_H370

Examples:

has_H225
  Indicates highly flammable liquid and vapor.

has_H319
  Indicates serious eye irritation.

has_H336
  Indicates possible drowsiness or dizziness.

has_H370
  Indicates possible organ damage.

These columns help the model understand safety concerns connected to each
solvent.


9. Why We Use These Columns
---------------------------

We use these columns because solvent recommendation is not based on the solvent
name only.

The system needs to understand:

- the solvent properties
- the catalyst
- the catalyst role
- the temperature
- the solvent guide color
- the hazard codes
- known incompatibilities
- whether the reaction is heated
- whether the catalyst role makes sense

Together, these features help the model learn patterns such as:

- some solvents are safer or greener than others
- some solvents are not stable under strong acid or strong base
- heating can increase risk
- a catalyst must support the role selected by the user
- hazard codes are important for risk comparison


10. Model Used
--------------

The project uses a RandomForestClassifier.

This model was chosen because:

- it works well for tabular datasets
- it can handle numeric and binary features
- it is easy to train for a prototype
- it gives strong baseline performance without needing a very large dataset

The model is trained from:

app/ml/training_data.csv

The trained model is saved as:

app/ml/greenchem_model.joblib


11. Important Limitation
------------------------

This dataset is not experimental laboratory proof.

It is generated from:

- the local solvent/catalyst knowledge base
- compatibility rules
- curated bootstrap logic

This means the model is useful for decision support, screening, and demo
purposes, but it cannot replace laboratory validation.

The application must always inform the user:

"This is an AI-assisted recommendation. The final decision must be made by the
user or a qualified human expert. Wet lab validation is required."


12. Short Pitch Explanation
---------------------------

Our dataset contains 7,800 rows and 28 columns. It is generated from 65
solvents, 10 catalysts, 20 catalyst roles, and 6 temperature conditions. Each
row represents a possible solvent-catalyst-role-temperature combination.

The model learns to predict whether the combination is compatible or not. It
uses features such as boiling point, CHEM21 solvent guide ranking, GHS hazard
codes, known incompatibilities, catalyst role, and whether the reaction is
heated.

This allows GreenChem Navigator to suggest greener solvent alternatives while
explaining the safety and compatibility reasons behind each recommendation.
The result is not a final lab decision; it is AI-assisted decision support that
must be validated by a human expert.
